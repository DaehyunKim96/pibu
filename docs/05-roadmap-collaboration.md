# 05 · 함께 발전시키는 방법 (협업 & 로드맵)

> 요청: "서비스를 어떻게 함께 수정·발전시켜 나가면 좋을지 리서치해서 알려줘."
> 이 문서는 **우리 둘(사용자 + Claude)이 이 서비스를 키워가는 방식**에 대한 제안이다.

---

## A. 우리가 일하는 방식 (협업 루프)

작은 단위로 **만들고 → 보고 → 고치는** 루프를 빠르게 도는 것을 제안한다.

```
①  당신: 방향/우선순위 결정 ("이번엔 진단 정확도를 높이자")
②  나:   리서치 + 변경안 제시 (docs에 근거 기록)
③  나:   구현 → 빌드 검증 → 화면으로 보여줌
④  당신: 피드백 ("이 카피는 빼고, 비교 기능 추가")
⑤  반복
```

### 권장 작업 단위
- **한 번에 하나의 주제**만. (성분 데이터 / 진단 로직 / 디자인 중 하나)
- 큰 기능은 "초안 → 피드백 → 다듬기" 3타로 쪼갠다.
- 모든 판단은 `docs/decision-log.md`에 1줄이라도 남긴다 → 나중에 "왜 이렇게 했지?"를 추적.

### 나에게 일을 주는 좋은 방법 (예시)
- 🔧 데이터: *"올리브영 진정 카테고리 베스트 10개 추가하고 성분 매핑해줘"*
- 🧪 로직: *"진단에서 복합성으로 너무 자주 빠져. 임계값 조정안 2개 비교해줘"*
- 🎨 UX: *"제품 2개를 나란히 비교하는 화면 만들어줘"*
- 📚 신뢰: *"각 논문에 PubMed 링크 달고 출처 페이지 만들어줘"*

> 팁: "여러 안을 비교해줘"라고 하면 내가 선택지를 표로 제시하고 추천을 단다.

## C. 코드/데이터를 안전하게 키우는 규칙

이 구조는 **데이터만 늘려도 전 페이지가 자동 반영**되도록 설계돼 있다. 키울 때:

1. **새 제품**을 추가하기 전에, 거기 쓰인 **성분이 `ingredients.ts`에 있는지** 먼저 확인/추가.
2. 성분 추가 시 4종 세트 필수: 작용기전 · 부여 특성 · 민감 적합도 · 근거 1편↑.
3. `bestFor`는 7개 타입 id만. 타입을 늘리려면 `skinTypes.ts` + 진단 로직 함께 수정.
4. 변경 후 항상 `npm run build`로 타입 검증. (id 오타는 빌드가 잡아준다)
5. 의미 있는 판단은 `decision-log.md`에 기록.

## D. 리스크 & 사전 합의 사항

| 리스크 | 대응 |
| --- | --- |
| **의료적 책임** | "정보 제공, 진단 아님" 일관 유지 + 위험 신호 시 전문의 권고 |
| **크롤링 법적 이슈** | 공개 데이터/약관 검토, 제휴 API 우선 고려 |
| **성분 정보 오류** | 근거 필수화 + 사람 검수 + 출처 노출로 정정 가능하게 |
| **과신 유발** | "경향 안내"임을 UI에 명시, 패치테스트 권고 |

## E. 지금 당장 정하면 좋은 것 (당신의 결정 필요)

다음 중 **어디부터 함께 다듬을지** 정해주면 거기서 시작한다:

1. **신뢰 강화** — 논문 실제 링크 + 진단 정확도 (가장 추천)
2. **데이터 확장** — 제품 수 늘리기 / 크롤링 설계
3. **기능 추가** — 루틴 빌더 · 제품 비교 · 찜
4. **디자인 다듬기** — 특정 화면 사용성 개선
