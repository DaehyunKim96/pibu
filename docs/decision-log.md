# 의사결정 기록 (Decision Log / ADR)

> 주요 판단과 그 이유를 시간순으로 남긴다. 형식: 결정 / 이유 / 대안 / 재검토 시점.

---

### 2026-06-20 · 기술 스택을 Vite + React + TS로 결정
- **결정**: 정적 사이트 대신 React SPA.
- **이유**: 진단 퀴즈·다중 필터·상세 토글 등 인터랙션 비중이 크고, 데이터(성분/제품/타입)를
  id로 연결해 재사용하기 좋음. TS로 id 오타를 빌드 단계에서 차단.
- **대안**: 순수 HTML/JS(인터랙션 한계), Next.js(SSR 필요성 아직 낮아 과함).
- **재검토**: SEO/공유가 중요해지면 Next.js 전환 검토.

### 2026-06-20 · 데이터를 코드 내 타입 객체로 (DB 아님)
- **결정**: `src/data/*.ts`에 큐레이션 데이터를 둠.
- **이유**: 초안 단계 속도 + 타입 안전. 구조만 맞으면 추후 DB로 무손실 이전 가능.
- **재검토**: 제품 100개↑ 또는 사용자 생성 데이터 도입 시 백엔드+DB.

### 2026-06-20 · 제품 데이터는 "크롤링 대신 큐레이션"으로 시작
- **결정**: 실시간 크롤링 보류, 공개 전성분 기반 수기 데이터셋.
- **이유**: 크롤링은 약관/법적 검토 필요. 초안에선 **구조와 UX 검증이 우선**.
  구조를 소스 교체만으로 크롤링 전환 가능하게 설계함.
- **재검토**: v0.3에서 파이프라인화.

### 2026-06-20 · 피부 타입을 Baumann 참고 7종으로 압축
- **결정**: 16종 전체 대신 민감·트러블 중심 7 아키타입.
- **이유**: pibu 페르소나에 맞는 실용성. 16종은 사용자 인지 부담↑.
- **대안**: 3분류(너무 거침), Baumann 16종(복잡).
- **재검토**: 사용자 데이터로 임계값/세분화 보정.

### 2026-06-20 · 진단 분기에서 "장벽 손상"을 최우선으로
- **결정**: barrier ≥ 3 이면 다른 축보다 먼저 '장벽 손상'으로 분류.
- **이유**: 장벽이 무너진 사람에게 활성(BHA 등)을 권하면 해롭다 → 안전 우선.

### 2026-06-20 · 신뢰를 핵심 자산으로 — 근거·주의 성분 동등 노출
- **결정**: 효과뿐 아니라 주의 성분·논문 근거·면책을 같은 비중으로 표시.
- **이유**: 민감 피부 사용자에게 '과장 없음'이 곧 차별점이자 브랜드 신뢰.

### 2026-06-20 · "~성분 때문에 ~특성" 을 데이터 구조로 강제
- **결정**: 제품마다 `highlights: {ingredientId, effect}` 필수.
- **이유**: 사용자 요구(근거형 설명)를 자유 텍스트가 아닌 **구조**로 보장 → 일관성·재사용.
