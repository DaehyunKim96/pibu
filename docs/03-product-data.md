# 03 · 제품 데이터 수집 방법론

> 코드: `src/data/products.ts`

## 데이터 출처 & 정직한 한계

- 현재 21개 제품은 **실재하는 대표 제품들의 공개 전성분**을 바탕으로 한 **큐레이션 데이터셋**이다.
- 실시간 크롤링이 아니라 **수기 정제 데이터**다. (이유: 초안 단계에서 구조·UX 검증이 우선,
  크롤링은 법적/안정성 검토가 필요 → `05-roadmap`)
- 가격·용량은 시점에 따라 변동될 수 있는 **근사치**다.

## 선정 기준

올리브영 등에서 **민감·트러블 피부가 실제로 많이 찾는** 카테고리 균형을 맞춰 선정:

- 토너/세럼/크림/선케어/각질케어/마스크 전 카테고리 커버
- 진정(병풀·어성초), 장벽(세라마이드), 활성(BHA·레티놀·아젤라산), 미백(비타민C) 등 **성분 다양성**
- 더마 브랜드(라로슈포제·에스트라)부터 가성비(일리윤)까지 **가격대 분산**

## 제품 1건의 데이터 구성

```ts
{
  brand, name, category, priceKRW, volume,
  keyIngredients: ['ceramide', ...]   // → ingredients.ts id 로 연결 (단일 출처)
  cautionIngredients: ['essentialoil'] // 있으면 상세에서 경고
  traits: ['장벽 강화', '보습']        // 성분에서 도출한 특성
  keywords: ['#세라마이드', ...]        // 해시태그
  bestFor: ['dry-sensitive', ...]      // → skinTypes.ts id
  highlights: [{ingredientId, effect}] // "○○ 성분 때문에 ○○하다" 근거 ★
  detail, fragranceFree
}
```

### 핵심 설계: `highlights`
사용자 요구사항인 **"~성분 때문에 ~적인 특성을 가진다"** 를 구조적으로 강제하는 필드.
모든 제품은 성분 id와 그 효과 문장을 짝지어 갖는다 → 상세 페이지 "왜 이런 특성을 가질까?" 섹션에 렌더.

## 특성·키워드 도출 규칙

1. 제품의 `keyIngredients`를 펼쳐 각 성분의 `confersTraits`를 모은다.
2. 대표 특성 3~4개를 `traits`로 선별 (제품 컨셉 우선순위 반영).
3. 키워드(`keywords`)는 검색·필터를 고려해 #성분 / #고민 / #속성(무향 등) 조합.
4. `bestFor`는 성분 궁합 + 제형(유분감)으로 결정. 예) 산(acid) 제품은 장벽손상 타입에서 제외.

## 데이터 무결성 원칙

- 제품의 모든 `keyIngredients`/`cautionIngredients`는 **반드시 `ingredients.ts`에 존재**해야 함
  (없는 id는 UI에서 자동 무시되지만, 추가 시 성분부터 등록하는 게 원칙).
- `bestFor`는 `skinTypes.ts`의 7개 id만 사용.
- 무향 여부(`fragranceFree`)는 향료/정유 미포함을 의미 → 민감 필터의 근거.

## 크롤링 전환 설계 (다음 단계)

현재 구조는 그대로 두고 **데이터 소스만 교체** 가능:

```
[수집] 올리브영/브랜드몰 전성분 → [정규화] INCI 표준화 → [매핑] 성분 id 연결
   → [LLM 보조] 특성/키워드 초안 생성 → [사람 검수] → products.ts(또는 DB)
```

→ 코드 변경 없이 `PRODUCTS` 배열만 채우면 전 페이지가 작동한다.
